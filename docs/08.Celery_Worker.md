# Celery Worker — Knowledge Transfer

## What is it?

The Celery Worker handles **long-running background tasks** that would be too slow for a synchronous API request. Its primary jobs:
1. **Fetch SEC EDGAR filings** — Download, parse HTML, chunk text, store in Postgres
2. **Compute DCF** — Run discounted cash flow models in the background
3. **Orchestrate analysis** — Queue filing fetches for a stock

---

## Technology Deep Dive: Celery

### What is Celery?

[Celery](https://docs.celeryq.dev/) is a **distributed task queue** for Python. It allows you to:
- Define tasks (Python functions) that run **asynchronously** in worker processes
- Queue tasks via a **message broker** (Redis in this project)
- Store results in a **backend** (also Redis)
- Scale by adding more worker processes

### How Celery Works

```
Producer (any service)          Redis (Broker)           Celery Worker
    │                              │                        │
    │ task.delay("NVDA", "10-K")   │                        │
    │──────────────────────────►   │                        │
    │                              │  Message in queue       │
    │                              │───────────────────────► │
    │                              │                         │ Execute task
    │                              │                         │ (fetch EDGAR,
    │                              │                         │  parse HTML,
    │                              │                         │  store chunks)
    │                              │   ◄─────────────────── │
    │                              │  Result stored           │
    │  ◄───────────────────────── │                          │
    │  (poll for result)           │                          │
```

**Key concepts**:

| Concept | What it is | In this project |
|---------|-----------|-----------------|
| **Broker** | Message queue that delivers tasks to workers | Redis (`redis://redis:6379`) |
| **Backend** | Store where task results are saved | Redis (same instance) |
| **Task** | A decorated Python function that runs in a worker | `@celery_app.task(name="fetch_edgar_filing")` |
| **Worker** | A long-running process that consumes tasks from the queue | The Docker container |
| `.delay()` | Non-blocking call that queues a task for execution | `fetch_edgar_filing.delay("NVDA", "10-K")` |
| **Concurrency** | Number of tasks a worker can run simultaneously | 2 (Raspberry Pi constraint) |

### Why Celery Instead of asyncio?

The fundamental agent already uses `asyncio`. Why not use that for background tasks?

| asyncio | Celery |
|---------|--------|
| Single process, single thread | Multi-process (isolates memory) |
| Async I/O only (no blocking code) | Can run blocking code (psycopg2, BeautifulSoup) |
| Task dies if the service restarts | Tasks persist in Redis queue across restarts |
| No retry/scheduling built-in | Built-in retry, rate limiting, scheduling |
| Good for: fast I/O tasks | Good for: CPU-heavy, slow, fallible tasks |

EDGAR fetching is slow (HTTP to SEC.gov, parsing large HTML documents). Celery isolates this from the main API.

## Key Components

### 1. Celery Configuration

[**Code: main.py:12-22**](https://github.com/utkarshmahar/stocks/blob/main/services/worker/main.py#L12)

```python
celery_app = Celery("worker", broker=settings.redis_url, backend=settings.redis_url)
celery_app.conf.update(
    task_serializer="json",
    result_serializer="json",
    accept_content=["json"],
    timezone="UTC",
    enable_utc=True,
    worker_max_memory_per_child=200000,  # 200MB
)
```

**Configuration breakdown**:

| Setting | Value | Why |
|---------|-------|-----|
| `broker` | `redis://redis:6379` | Redis receives task messages |
| `backend` | `redis://redis:6379` | Redis stores task results |
| `task_serializer` | `json` | Tasks are serialized as JSON (not pickle — more secure) |
| `worker_max_memory_per_child` | 200,000 KB (200MB) | Worker process is killed and restarted if it exceeds 200MB — prevents memory leaks on the Pi |

**Why JSON serializer?** Pickle (Python's default serializer) can execute arbitrary code during deserialization — a security risk if the Redis broker is compromised. JSON is safe.

### 2. Sync Postgres Connection

[**Code: main.py:24-35**](https://github.com/utkarshmahar/stocks/blob/main/services/worker/main.py#L24)

```python
SYNC_POSTGRES_URL = settings.postgres_url.replace("postgresql+asyncpg://", "postgresql://")

def get_sync_connection():
    import psycopg2
    return psycopg2.connect(host=host, port=port, dbname=db, user=user, password=password)
```

**Why sync instead of async?**

Celery workers run in **forked processes** (using `prefork` pool). Python's `asyncio` event loop doesn't survive `fork()` — you'd get "event loop is closed" errors. So the worker uses:
- `psycopg2` (synchronous) instead of `asyncpg` (asynchronous)
- `httpx` in synchronous mode instead of async mode

The URL conversion replaces `postgresql+asyncpg://` (SQLAlchemy async dialect) with plain `postgresql://` (psycopg2 dialect).

### 3. EDGAR Filing Fetch Task

[**Code: main.py:38-81**](https://github.com/utkarshmahar/stocks/blob/main/services/worker/main.py#L38)

```python
@celery_app.task(name="fetch_edgar_filing")
def fetch_edgar_filing(symbol: str, filing_type: str = "10-K"):
```

**What is EDGAR?**

EDGAR (Electronic Data Gathering, Analysis, and Retrieval) is the SEC's system where all public companies file financial documents:
- **10-K**: Annual report (full financials, risk factors, business description)
- **10-Q**: Quarterly report (shorter, more frequent updates)

**The fetch process**:

1. **Search EDGAR** for the company's filings:
```python
cik_resp = httpx.get(
    f"https://www.sec.gov/cgi-bin/browse-edgar?action=getcompany&company={symbol}&type={filing_type}",
    headers={"User-Agent": "OptionsTrader research@example.com"},
)
```
The User-Agent is required — SEC blocks requests without it.

2. **Parse the filing index page** to find filing document URLs:
```python
soup = BeautifulSoup(cik_resp.text, "html.parser")
links = soup.find_all("a", href=re.compile(r"/Archives/edgar/data"))
```

3. **Download the actual filing**:
```python
filing_resp = httpx.get(filing_url, headers=headers, timeout=60.0, follow_redirects=True)
```
60-second timeout because 10-K filings can be large (10+ MB of HTML).

4. **Extract text from HTML**:
```python
text_content = BeautifulSoup(filing_resp.text, "html.parser").get_text()
```
BeautifulSoup strips all HTML tags, leaving plain text.

5. **Chunk and store**:
```python
chunks = chunk_text(text_content, chunk_size=1000)
store_chunks(symbol, filing_type, chunks)
```

### 4. Text Chunking

[**Code: main.py:84-101**](https://github.com/utkarshmahar/stocks/blob/main/services/worker/main.py#L84)

```python
def chunk_text(text: str, chunk_size: int = 1000) -> list[str]:
    text = re.sub(r'\s+', ' ', text).strip()  # Normalize whitespace
    words = text.split()
    chunks = []
    current = []
    current_len = 0
    for word in words:
        current.append(word)
        current_len += len(word) + 1
        if current_len >= chunk_size:
            chunks.append(" ".join(current))
            current = []
            current_len = 0
    if current:
        chunks.append(" ".join(current))
    return chunks
```

**Why chunk?**
- LLMs have context limits — can't feed a 500-page 10-K all at once
- Embeddings work better on focused chunks (~1000 characters) than full documents
- pgvector can search individual chunks for relevant sections

**Chunk size = 1000 characters** (~150-200 words). This is a good balance:
- Small enough for embeddings to capture focused meaning
- Large enough to contain complete paragraphs and context

### 5. Chunk Storage

[**Code: main.py:104-116**](https://github.com/utkarshmahar/stocks/blob/main/services/worker/main.py#L104)

```python
def store_chunks(symbol: str, filing_type: str, chunks: list[str]):
    conn = get_sync_connection()
    cur = conn.cursor()
    for i, chunk in enumerate(chunks):
        cur.execute(
            """INSERT INTO filing_embeddings (symbol, filing_type, filing_date, chunk_index, chunk_text)
               VALUES (%s, %s, %s, %s, %s)""",
            (symbol, filing_type, datetime.utcnow().date(), i, chunk),
        )
    conn.commit()
```

**Note**: The `embedding` column is left NULL here. A future enhancement would:
1. Send each chunk to an embedding API (e.g., Anthropic, OpenAI, or a local model)
2. Store the resulting vector in the `embedding` column
3. Use vector similarity search in the fundamental agent

### 7. Analysis Orchestrator

[**Code: main.py:158-164**](https://github.com/utkarshmahar/stocks/blob/main/services/worker/main.py#L158)

```python
@celery_app.task(name="run_fundamental_analysis")
def run_fundamental_analysis(symbol: str):
    fetch_edgar_filing.delay(symbol, "10-K")
    fetch_edgar_filing.delay(symbol, "10-Q")
    return {"status": "analysis_queued", "symbol": symbol}
```

**Task chaining**: This task doesn't do analysis itself — it queues two EDGAR fetch tasks (10-K and 10-Q). These run in parallel since they're independent `.delay()` calls.

**Note**: The actual AI analysis is not yet triggered from this task. In a complete implementation, it would chain:
1. Fetch 10-K → store chunks
2. Fetch 10-Q → store chunks
3. Run fundamental agent analysis (with filing context now available)

## Technology Deep Dive: BeautifulSoup

[BeautifulSoup](https://beautiful-soup-4.readthedocs.io/) is a Python HTML/XML parser. In this project:

```python
from bs4 import BeautifulSoup

# Parse HTML page
soup = BeautifulSoup(html_text, "html.parser")

# Find specific elements
links = soup.find_all("a", href=re.compile(r"/Archives/edgar/data"))

# Extract all text (strip HTML tags)
text_content = soup.get_text()
```

**Why BeautifulSoup for EDGAR?** SEC filings are delivered as raw HTML with tables, headers, and formatting. BeautifulSoup strips all markup to get clean text for chunking and analysis.

## Deployment

[**Code: Dockerfile**](https://github.com/utkarshmahar/stocks/blob/main/services/worker/Dockerfile)

```dockerfile
CMD ["celery", "-A", "main.celery_app", "worker", "--loglevel=info", "--concurrency=2"]
```

| Flag | Meaning |
|------|---------|
| `-A main.celery_app` | Use the Celery app defined in `main.py` |
| `worker` | Run as a worker (consumer) |
| `--loglevel=info` | Log task execution at info level |
| `--concurrency=2` | Run 2 task processes simultaneously |

**Why concurrency=2?** The Raspberry Pi has 4 cores but limited RAM. 2 concurrent Celery workers (each potentially using 200MB) keeps memory usage manageable.

- **No port exposed** — the worker doesn't serve HTTP; it only consumes from Redis
- Memory limit: **512MB** (largest service — EDGAR documents can be large)
- Runs under the `full` Docker Compose profile

| `beautifulsoup4` | HTML parsing for SEC filings |
| `sqlalchemy[asyncio]` + `asyncpg` | ORM (used for config loading) |

| `pgvector` | Vector extension bindings |
| `pydantic-settings` | Config from environment |

**Why both `asyncpg` and `psycopg2`?** The shared config module uses SQLAlchemy async (which needs asyncpg). But Celery tasks use psycopg2 directly because Celery's prefork pool doesn't support asyncio.
