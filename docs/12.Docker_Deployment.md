# Docker & Deployment — Knowledge Transfer

## What is it?

The platform runs as a **multi-container Docker Compose application** on a Raspberry Pi 5 (16GB). Docker Compose orchestrates 10+ containers with health checks, dependency ordering, memory limits, and shared networking.

---

## Technology Deep Dive: Docker & Docker Compose

### Docker Concepts

| Concept | What it is | In this project |
|---------|-----------|-----------------|
| **Image** | A read-only template with code + dependencies | `python:3.11-slim`, `redis:7-alpine`, custom service images |
| **Container** | A running instance of an image | One container per service |
| **Dockerfile** | Instructions to build an image | `FROM`, `COPY`, `RUN`, `CMD` |
| **Volume** | Persistent storage that survives container restarts | `postgres_data`, `redis_data` |
| **Network** | Virtual network connecting containers | Default compose network (all services connected) |
| **Health check** | Periodic test to verify container is working | `pg_isready`, `redis-cli ping` |
| **Profile** | Grouping mechanism to start subsets of services | `default` (core) vs `full` (all services) |

### Docker Compose Concepts

Docker Compose is a tool for defining and running multi-container applications with a single YAML file.

```yaml
version: "3.8"           # Compose file format version
services:                 # Container definitions
  postgres:               # Service name (also DNS name inside the network)
    image: pgvector/pgvector:pg16  # Docker Hub image to use
    ports: ["5432:5432"]  # host:container port mapping
    volumes: [...]        # Persistent storage
    healthcheck: {...}    # Is the service ready?
    deploy:
      resources:
        limits:
          memory: 256M    # OOM kill at 256MB
```

### YAML Anchors (DRY Configuration)

```yaml
x-common: &common
  restart: unless-stopped
  extra_hosts:
    - "host.docker.internal:host-gateway"
  env_file:
    - .env
```

**`x-common: &common`**: Defines a reusable block. Services reference it with `<<: *common` to inherit all properties.

| Property | What it does |
|----------|-------------|
| `restart: unless-stopped` | Auto-restart on crash (but not if manually stopped) |
| `extra_hosts: host.docker.internal:host-gateway` | Makes `host.docker.internal` resolve to the host machine's IP |
| `env_file: .env` | Loads environment variables from `.env` file |

**Why `host.docker.internal`?** InfluxDB and Grafana run natively on the Pi (not in Docker). Containers need a way to reach the host. This special hostname bridges the gap.

### Infrastructure Services

#### PostgreSQL (pgvector)

```yaml
postgres:
  image: pgvector/pgvector:pg16
  environment:
    POSTGRES_USER: ${POSTGRES_USER}
    POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    POSTGRES_DB: ${POSTGRES_DB}
  ports: ["5432:5432"]
  volumes:
    - postgres_data:/var/lib/postgresql/data
    - ./migrations/001_initial_schema.sql:/docker-entrypoint-initdb.d/001_initial_schema.sql
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER}"]
    interval: 10s
    timeout: 5s
    retries: 5
  deploy:
    resources:
      limits:
        memory: 256M
```

**Key details**:
- **Image**: `pgvector/pgvector:pg16` — PostgreSQL 16 with pgvector extension pre-installed
- **Volumes**:
  - `postgres_data` — persistent data (survives `docker compose down`)
  - `001_initial_schema.sql` mounted into `/docker-entrypoint-initdb.d/` — automatically executed on first startup (creates all tables)
- **Health check**: `pg_isready` is a PostgreSQL utility that checks if the server is accepting connections
- **Environment variables**: `${POSTGRES_USER}` reads from `.env` file

#### Redis

```yaml
redis:
  image: redis:7-alpine
  ports: ["6379:6379"]
  volumes: [redis_data:/data]
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
  deploy:
    resources:
      limits:
        memory: 128M
```

**Alpine image**: `redis:7-alpine` is a minimal Linux distribution (~5MB). The full Redis image would be ~130MB. On a Pi with limited storage, every MB matters.

**Health check**: `redis-cli ping` sends a PING command. Redis responds with PONG if healthy.

### Core Services (Default Profile)

These start with `docker compose up -d`:

```yaml
ingestion-service:
  <<: *common
  build:
    context: .
    dockerfile: services/ingestion-service/Dockerfile
  volumes:
    - ./schwab_token.json:/app/schwab_token.json:ro
  depends_on:
    postgres: { condition: service_healthy }
    redis: { condition: service_healthy }
  deploy:
    resources:
      limits:
        memory: 256M
```

**Build context**: `context: .` means Docker uses the repo root as the build context. This allows Dockerfiles to `COPY shared/ /app/shared/` from the root.

**Volume mount**: `schwab_token.json:/app/schwab_token.json:ro` — mounts the Schwab OAuth token as read-only inside the container.

**`depends_on` with `service_healthy`**: The ingestion service only starts after PostgreSQL and Redis pass their health checks. This prevents startup crashes from missing dependencies.

```yaml
api-gateway:
  <<: *common
  ports: ["8000:8000"]
  depends_on:
    postgres: { condition: service_healthy }
    redis: { condition: service_healthy }

frontend:
  <<: *common
  build:
    context: ./frontend
    dockerfile: Dockerfile
  ports: ["3001:80"]
  depends_on: [api-gateway]
```

**Frontend**: Maps port 3001 (host) → 80 (container/nginx). The frontend depends on the API gateway being up.

### Full Profile Services

These start with `docker compose --profile full up -d`:

```yaml
quant-engine:
  <<: *common
  profiles: ["full"]
  deploy:
    resources:
      limits:
        memory: 384M

options-agent:
  <<: *common
  profiles: ["full"]

portfolio-service:
  <<: *common
  profiles: ["full"]
  volumes:
    - ./schwab_token.json:/app/schwab_token.json:ro

risk-engine:
  <<: *common
  profiles: ["full"]
  deploy:
    resources:
      limits:
        memory: 128M

fundamental-agent:
  <<: *common
  profiles: ["full"]
  deploy:
    resources:
      limits:
        memory: 384M

worker:
  <<: *common
  profiles: ["full"]
  deploy:
    resources:
      limits:
        memory: 512M
```

**Profiles**: The `full` profile groups AI agents, quant engine, portfolio service, risk engine, and worker. This allows running just the data pipeline (core) when you don't need AI features — saves ~2GB of RAM on the Pi.

### Named Volumes

```yaml
volumes:
  postgres_data:
  redis_data:
```

**Named volumes** persist data across container restarts and `docker compose down`. Only `docker compose down -v` or `docker volume rm` deletes them.

- `postgres_data`: All PostgreSQL data (tables, indexes, WAL)
- `redis_data`: Redis RDB snapshots (persisted to disk periodically)

## Memory Budget

The Raspberry Pi 5 has 16GB RAM. Here's the memory allocation:

| Service | Memory Limit | Notes |
|---------|-------------|-------|
| PostgreSQL | 256MB | Shared buffers + connections |
| Redis | 128MB | In-memory data + pub/sub buffers |
| Ingestion | 256MB | Schwab responses + InfluxDB writes |
| API Gateway | 256MB | WebSocket connections + HTTP handling |
| Frontend (nginx) | 128MB | Static file serving |
| Quant Engine | 384MB | NumPy arrays + InfluxDB query results |
| Options Agent | 256MB | Claude API responses |
| Portfolio | 256MB | Schwab API responses |
| Risk Engine | 128MB | Pure validation logic |
| Fundamental Agent | 384MB | Claude responses + filing text |
| Worker (Celery) | 512MB | EDGAR documents (large HTML) |
| **Total (full)** | **~2.9GB** | Leaves ~13GB for OS, InfluxDB, Grafana |

**Note**: These are Docker `memory` limits. The kernel OOM-kills the container if it exceeds the limit. The actual usage is typically 30-60% of the limit.

## Networking

All services share a default Docker Compose network. Service names are DNS hostnames:

```
ingestion-service → http://ingestion-service:8010
api-gateway       → http://api-gateway:8000
quant-engine      → http://quant-engine:8020
options-agent     → http://options-agent:8030
portfolio-service → http://portfolio-service:8040
risk-engine       → http://risk-engine:8050
fundamental-agent → http://fundamental-agent:8060
postgres          → postgres:5432
redis             → redis:6379
```

**Only externally accessible**:
- Port 8000: API Gateway (for direct API access)
- Port 3001: Frontend (nginx serves the dashboard)
- Port 5432: PostgreSQL (for debugging with psql/pgAdmin)
- Port 6379: Redis (for debugging with redis-cli)

## Common Docker Operations

```bash
# Start core services
sudo docker compose up -d

# Start all services
sudo docker compose --profile full up -d

# View logs
sudo docker compose logs -f ingestion-service
sudo docker compose logs -f --tail=50 options-agent

# Rebuild a single service
sudo docker compose build ingestion-service
sudo docker compose up -d ingestion-service

# Rebuild all
sudo docker compose build
sudo docker compose --profile full up -d

# Stop everything
sudo docker compose --profile full down

# Stop + delete data
sudo docker compose --profile full down -v

# Check status
sudo docker compose ps

# Shell into a container
sudo docker compose exec api-gateway bash

# View resource usage
sudo docker stats
```
