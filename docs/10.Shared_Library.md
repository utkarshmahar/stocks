# Shared Library — Knowledge Transfer

## What is it?

The `shared/` directory is a **Python package** that provides common utilities used by all backend services. Instead of duplicating database clients, configuration classes, and models across 8 services, they're centralized here and copied into each Docker image at build time.

---

## File Structure

```
shared/
├── __init__.py          # Makes it a Python package
├── config.py            # Pydantic settings (env vars → typed config)
├── health.py            # FastAPI /health endpoint router
├── influxdb_client.py   # InfluxDB read/write via HTTP API
├── models.py            # Pydantic data models (shared schema)
├── postgres.py          # SQLAlchemy async engine + session factory
├── redis_client.py      # Redis async client singleton
└── schwab_auth.py       # Schwab OAuth2 token-based authentication
```

## Components

### 1. Configuration (`config.py`)

[**Code: shared/config.py**](https://github.com/utkarshmahar/stocks/blob/main/shared/config.py)

```python
class Settings(BaseSettings):
    schwab_api_key: str = ""
    schwab_app_secret: str = ""
    influxdb_url: str = "http://host.docker.internal:8086"
    influxdb_token: str = ""
    influxdb_org: str = "options_trading"
    influxdb_bucket: str = "options_db"
    postgres_url: str = "postgresql+asyncpg://trading:changeme@postgres:5432/trading"
    redis_url: str = "redis://redis:6379"
    anthropic_api_key: str = ""
    query_interval: int = 30
    strike_count: int = 10
    default_symbols: str = "PANW,QCOM,CSCO"

    model_config = {"env_file": ".env", "extra": "ignore"}

@lru_cache
def get_settings() -> Settings:
    return Settings()
```

#### Technology: Pydantic Settings

`pydantic-settings` automatically loads configuration from **environment variables**:

1. Reads `.env` file (if present)
2. Reads actual environment variables (override `.env`)
3. Falls back to default values
4. **Type-coerces** automatically: `query_interval: int = 30` converts the string env var `QUERY_INTERVAL=30` to an integer

**`@lru_cache`**: The `get_settings()` function is cached — it reads environment variables once and reuses the same `Settings` object for the lifetime of the process. This prevents re-reading the `.env` file on every call.

**`extra = "ignore"`**: Unknown environment variables are silently ignored. Without this, any extra env var would raise a validation error.

**`host.docker.internal`**: This special hostname resolves to the host machine's IP from inside a Docker container. Used for InfluxDB, which runs natively (not in Docker).

### 3. InfluxDB Client (`influxdb_client.py`)

[**Code: shared/influxdb_client.py**](https://github.com/utkarshmahar/stocks/blob/main/shared/influxdb_client.py)

#### Write Function

```python
async def influx_write(lines: list[str]) -> bool:
    url = f"{settings.influxdb_url}/api/v2/write?org={settings.influxdb_org}&bucket={settings.influxdb_bucket}&precision=ns"
    headers = {
        "Authorization": f"Token {settings.influxdb_token}",
        "Content-Type": "text/plain",
    }
    payload = "\n".join(lines)
    async with httpx.AsyncClient(timeout=30.0) as client:
        resp = await client.post(url, content=payload, headers=headers)
        return resp.status_code == 204
```

**Why raw HTTP instead of the official InfluxDB Python client?**
- The official client (`influxdb-client-python`) is heavy and has sync-only write APIs
- Raw HTTP with `httpx` is async-native and lightweight
- InfluxDB's write API is simple: POST line protocol text to `/api/v2/write`
- Status code 204 = success (no body returned)

**Batch writing**: All lines are joined with `\n` and sent in a single HTTP request. This is critical for performance — writing one point at a time would be 100x slower due to HTTP overhead.

#### Query Function

```python
async def influx_query(flux_query: str) -> list[dict]:
    url = f"{settings.influxdb_url}/api/v2/query?org={settings.influxdb_org}"
    headers = {
        "Authorization": f"Token {settings.influxdb_token}",
        "Content-Type": "application/vnd.flux",
        "Accept": "application/csv",
    }
    async with httpx.AsyncClient(timeout=30.0) as client:
        resp = await client.post(url, content=flux_query, headers=headers)
        return _parse_csv(resp.text)
```

**CSV response parsing**: InfluxDB returns query results as CSV (not JSON). The `_parse_csv()` function parses this into a list of dicts, skipping comment lines (`#`) and metadata columns.

### 4. PostgreSQL Client (`postgres.py`)

[**Code: shared/postgres.py**](https://github.com/utkarshmahar/stocks/blob/main/shared/postgres.py)

```python
def get_engine():
    global _engine
    if _engine is None:
        _engine = create_async_engine(
            settings.postgres_url,
            pool_size=5,
            max_overflow=2,
            pool_recycle=300,
        )
    return _engine

def get_session_factory() -> async_sessionmaker[AsyncSession]:
    global _session_factory
    if _session_factory is None:
        _session_factory = async_sessionmaker(get_engine(), expire_on_commit=False)
    return _session_factory

async def get_db() -> AsyncSession:
    factory = get_session_factory()
    async with factory() as session:
        yield session
```

#### Technology: SQLAlchemy Async + asyncpg

**SQLAlchemy** is Python's most popular ORM (Object-Relational Mapper). In this project, it's used in **async mode** with `asyncpg` as the database driver.

**Connection pooling**:
| Parameter | Value | Meaning |
|-----------|-------|---------|
| `pool_size=5` | 5 persistent connections | Pre-opened connections ready for use |
| `max_overflow=2` | 2 extra connections | Temporary connections when pool is full |
| `pool_recycle=300` | 5 minutes | Connections older than 5 min are recycled (prevents stale connections) |

**Why `pool_size=5` on a Pi?** PostgreSQL's default `max_connections` is 100. With 8 services × 5 connections = 40 connections max. Leaves headroom for psql, Grafana, etc.

**`expire_on_commit=False`**: After committing, SQLAlchemy normally expires all loaded objects (forces re-query on next access). Disabling this avoids unnecessary re-queries.

**Singleton pattern**: `_engine` and `_session_factory` are module-level globals. `get_engine()` creates the engine once and reuses it. This prevents creating a new connection pool per request.

**`get_db()` as a FastAPI dependency**:
```python
@app.get("/api/watchlist")
async def get_watchlist(db: AsyncSession = Depends(get_db)):
    result = await db.execute(text("SELECT ..."))
```

`Depends(get_db)` is a generator-based dependency:
1. `yield session` provides the session to the endpoint
2. After the endpoint returns, the `async with` block closes the session
3. This ensures sessions are always properly closed, even if the endpoint throws an exception

### 5. Redis Client (`redis_client.py`)

[**Code: shared/redis_client.py**](https://github.com/utkarshmahar/stocks/blob/main/shared/redis_client.py)

```python
import redis.asyncio as aioredis

_redis = None

async def get_redis() -> aioredis.Redis:
    global _redis
    if _redis is None:
        _redis = aioredis.from_url(settings.redis_url, decode_responses=True)
    return _redis
```

#### Technology: Redis

**Redis** (Remote Dictionary Server) is an in-memory data store used for three purposes in this project:

1. **Pub/Sub** (Publisher/Subscriber messaging):
   - Ingestion service publishes options data to `options:updates` channel
   - API gateway subscribes and forwards to WebSocket clients
   - This is **fire-and-forget** — if no one is subscribed, messages are lost

2. **Celery Broker**:
   - Celery uses Redis as its message queue
   - Tasks are serialized as JSON and stored in Redis lists
   - Workers pop tasks from the queue

3. **Celery Backend**:
   - Task results are stored in Redis keys
   - Allows producers to poll for results

**`decode_responses=True`**: Redis stores everything as bytes. This flag automatically decodes responses to Python strings.

**Singleton pattern**: Same as Postgres — one Redis connection is created and reused across the entire service lifetime. Redis connections are multiplexed (one TCP connection handles many concurrent operations).

### 6. Schwab Authentication (`schwab_auth.py`)

[**Code: shared/schwab_auth.py**](https://github.com/utkarshmahar/stocks/blob/main/shared/schwab_auth.py)

```python
def get_schwab_client():
    settings = get_settings()
    try:
        client = schwab.auth.client_from_token_file(
            settings.schwab_token_path,
            settings.schwab_api_key,
            settings.schwab_app_secret,
        )
        return client
    except FileNotFoundError:
        raise RuntimeError("Token file not found. Run authenticate_schwab.py first.")
    except Exception as e:
        if "refresh token" in str(e).lower() and "expired" in str(e).lower():
            raise RuntimeError("Schwab refresh token expired (7-day limit). Re-authenticate.")
        raise
```

**OAuth2 token lifecycle**:
```
User runs authenticate_schwab.py
    → Browser opens Schwab login page
    → User logs in
    → Schwab redirects with authorization code
    → Script exchanges code for access token + refresh token
    → Tokens saved to schwab_token.json

schwab-py SDK:
    → Reads tokens from file
    → Uses access token for API calls
    → When access token expires (~30 min), uses refresh token to get a new one
    → When refresh token expires (7 days), RuntimeError is raised
```

**Error handling**: Two specific failure modes are detected:
1. **Token file not found**: User hasn't run initial authentication
2. **Refresh token expired**: 7-day Schwab limit hit — manual re-auth required

### 7. Data Models (`models.py`)

[**Code: shared/models.py**](https://github.com/utkarshmahar/stocks/blob/main/shared/models.py)

All Pydantic models used across services:

| Model | Used by | Fields |
|-------|---------|--------|
| `OptionLeg` | Options Agent, Risk Engine | action, strike, type, expiry |
| `Recommendation` | Options Agent | symbol, strategy, legs, profit/loss, probability, reasoning |
| `QuantSummary` | Quant Engine | IV percentile, IV rank, skew, unusual activity |
| `PortfolioPosition` | Portfolio Service | quantity, avg_price, premium_collected, adjusted_cost |
| `StockQuote` | API Gateway | price, bid, ask, change, volume |
| `OptionContract` | API Gateway | strike, bid, ask, IV, Greeks, DTE |
| `OptionsChain` | API Gateway | underlying quote + calls[] + puts[] |
| `RiskDecision` | Risk Engine | approved, risk_notes, flags |
| `FundamentalReport` | Fundamental Agent | intrinsic_value, report_data, ai_narrative |

**Why Pydantic models?** They provide:
- Type validation (catches malformed data early)
- Serialization/deserialization (`.model_dump()` → dict, `Model(**data)` → object)
- API documentation (FastAPI auto-generates OpenAPI schemas from these)
- Default values for optional fields

## How Shared Code Gets Into Docker Containers

[**Code: services/ingestion-service/Dockerfile**](https://github.com/utkarshmahar/stocks/blob/main/services/ingestion-service/Dockerfile#L9)

```dockerfile
COPY shared/ /app/shared/
```

Every service's Dockerfile copies the `shared/` directory into `/app/shared/`. Since the Docker build context is the repo root (`.` in docker-compose), this path is accessible.

The services import from it as:
```python
from shared.config import get_settings
from shared.postgres import get_db
```

This works because `/app` is the `WORKDIR` and `shared/` is a Python package (has `__init__.py`).
