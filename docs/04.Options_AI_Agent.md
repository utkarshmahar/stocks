# Options AI Agent — Knowledge Transfer

## What is it?

The Options AI Agent is the **LLM-powered brain** of the platform. It receives structured quantitative data from the Quant Engine, reasons about it using Claude (Anthropic's LLM), and outputs structured JSON recommendations for high-probability options selling strategies.

**Critical design rule**: The AI does NOT compute Greeks, prices, or probabilities. It only **reasons** about pre-computed data. All math comes from the Quant Engine.

---

## High-Level Flow

```
Quant Engine (structured JSON)
    │
    ├─ IV percentile, skew, unusual activity
    │
API Gateway (options chain)
    │
    ├─ Top 5 calls/puts by volume
    │
Postgres (agent config)
    │
    ├─ Risk tolerance, DTE range, delta range
    │
    ▼
Options AI Agent
    │
    ├─ Builds prompt from all inputs
    ├─ Calls Claude Sonnet API
    ├─ Parses JSON response
    ├─ Validates against Pydantic model
    │
    ▼
Risk Engine (validation)
    │
    ▼
Postgres (save recommendation + audit log)
```

## Technology Deep Dive: Anthropic Claude API

### How the Claude API Works

[**Code: main.py:121-133**](https://github.com/utkarshmahar/stocks/blob/main/services/options-agent/main.py#L121)

```python
client = anthropic.Anthropic(api_key=settings.anthropic_api_key)
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=SYSTEM_PROMPT,
    messages=[{"role": "user", "content": user_prompt}],
)
content = response.content[0].text.strip()
rec_data = json.loads(content)
```

**The Anthropic SDK** (`anthropic` Python package) wraps the Claude REST API:

1. **Authentication**: API key passed at client creation
2. **Messages API**: Sends a conversation (system prompt + user message) to Claude
3. **Response**: Returns a `Message` object with `content[0].text` containing the LLM output

**Why `claude-sonnet-4-6`?** It's the mid-tier model — fast enough for real-time use (2-5 second responses), smart enough for structured reasoning. Opus would be better at nuance but slower and more expensive.

### System Prompt: Enforcing JSON Output

[**Code: main.py:22-45**](https://github.com/utkarshmahar/stocks/blob/main/services/options-agent/main.py#L22)

```python
SYSTEM_PROMPT = """You are an options trading AI agent specializing in high-probability income strategies.
You analyze structured quant data and recommend option selling strategies.

You MUST respond with ONLY valid JSON matching this exact schema:
{
  "symbol": "string",
  "strategy": "string (one of: Put Credit Spread, Call Credit Spread, Cash Secured Put, Covered Call, Iron Condor)",
  "legs": [{"action": "SELL|BUY", "strike": number, "type": "PUT|CALL", "expiry": "YYYY-MM-DD"}],
  "max_profit": number,
  "max_loss": number,
  "probability_estimate": number (0-1),
  "capital_required": number,
  "reasoning_summary": "string",
  "risk_flags": ["string"]
}

Rules:
- Focus on high-probability income strategies (>60% probability)
- Prefer 14-60 DTE unless data suggests otherwise
- Delta range 0.15-0.35 for short legs
- Always include risk flags for earnings, ex-div dates, high correlation
- Never recommend naked options
- Base probability on delta of short leg
"""
```

**Key design patterns**:

1. **Schema enforcement**: The system prompt contains the exact JSON schema. This is more reliable than asking for "structured output" in general.

2. **No markdown instruction**: `"You MUST respond with ONLY valid JSON"` prevents the LLM from wrapping output in ```json code blocks.

3. **Domain constraints**: Rules like "never naked options" and "delta 0.15-0.35" constrain the LLM to safe strategies.

4. **Probability grounding**: "Base probability on delta of short leg" tells the LLM to use a real metric rather than hallucinating a probability.

### User Prompt: Structured Data Input

[**Code: main.py:102-119**](https://github.com/utkarshmahar/stocks/blob/main/services/options-agent/main.py#L102)

```python
user_prompt = f"""Analyze this options data and recommend a high-probability sell strategy:

QUANT SUMMARY:
{json.dumps(quant, indent=2)}

OPTIONS CHAIN (sample - top 5 calls and puts by volume):
{json.dumps({...}, indent=2)}

AGENT CONFIG:
- Risk tolerance: {config.get('risk_tolerance', 'moderate')}
- DTE range: {config.get('dte_range', {"min": 14, "max": 60})}
- Delta range: {config.get('delta_range', {"min": 0.15, "max": 0.35})}
- Strategies enabled: {config.get('strategies_enabled', [...])}
"""
```

**Why top 5 by volume?** Sending the entire options chain would exceed token limits and add noise. The 5 most traded contracts represent where the market's attention is focused.

**Why include agent config?** The user can adjust risk tolerance and strategy preferences from the Settings page. The LLM adapts its recommendations accordingly.

```python
async def get_quant_summary(symbol: str) -> dict | None:

```

**Service-to-service communication**: The options agent calls other microservices via HTTP using Docker DNS names. This is the microservice pattern — each service has a single responsibility and exposes a REST API.

**Why httpx?** It's an async HTTP client (like `requests` but with `async/await`). This means the service can handle multiple concurrent requests without blocking.

### 2. JSON Parsing & Validation

[**Code: main.py:130-138**](https://github.com/utkarshmahar/stocks/blob/main/services/options-agent/main.py#L130)

```python
content = response.content[0].text.strip()
rec_data = json.loads(content)
rec = Recommendation(**rec_data)
return rec.model_dump()
```

**Three layers of validation**:
1. `json.loads()` — Ensures the LLM output is valid JSON (catches syntax errors)
2. `Recommendation(**rec_data)` — Pydantic model validates types, required fields, and constraints
3. If either fails, the function returns `None` and logs the error (graceful degradation)

**The `Recommendation` Pydantic model** ([shared/models.py](https://github.com/utkarshmahar/stocks/blob/main/shared/models.py)):
```python
class Recommendation(BaseModel):
    symbol: str
    strategy: str
    legs: list[OptionLeg]
    max_profit: float
    max_loss: float
    probability_estimate: float
    capital_required: float
    reasoning_summary: str
    risk_flags: list[str] = []
```

If the LLM returns `"max_profit": "four hundred"` instead of `400`, Pydantic will raise a `ValidationError` and the recommendation is discarded.

### 3. Risk Check Integration

[**Code: main.py:176-186**](https://github.com/utkarshmahar/stocks/blob/main/services/options-agent/main.py#L176)

```python
async def risk_check(rec: dict) -> dict:
    try:
        async with httpx.AsyncClient(timeout=10.0) as client:
            resp = await client.post("http://risk-engine:8050/api/risk/validate", json=rec)
            if resp.status_code == 200:
                return resp.json()
    except Exception:
        pass
    # Default: approve if risk engine is unavailable
    return {"approved": True, "risk_notes": "Risk engine unavailable — auto-approved", "flags": []}
```

**Graceful degradation**: If the risk engine is down, recommendations are auto-approved with a note. This prevents the system from going dark just because one service is unavailable. The note makes it transparent that risk validation was skipped.

### 5. Scheduled Analysis Loop

[**Code: main.py:218-250**](https://github.com/utkarshmahar/stocks/blob/main/services/options-agent/main.py#L218)

```python
async def scheduled_analysis():
    while True:
        now = datetime.utcnow()
        hour = now.hour
        if 14 <= hour < 21:  # Market hours: 9:30 AM - 4:00 PM ET (14:30 - 21:00 UTC)
            # Analyze all watchlist symbols
            ...
        else:
            print("[OPTIONS-AGENT] Market closed, skipping analysis.")
        await asyncio.sleep(300)  # 5 minutes
```

**How it works**:
- Background task spawned at startup via `asyncio.create_task()`
- Runs every 5 minutes during market hours (UTC 14:00-21:00)
- Outside market hours, it logs "market closed" and sleeps
- For each active symbol in the watchlist:
  1. Get quant summary
  2. Get options chain
  3. Call Claude API
  4. Run risk check
  5. Save to Postgres

**Why 5-minute intervals?** Options prices don't change second-by-second for selling strategies. 5 minutes gives enough data freshness while respecting Claude API rate limits and costs.
