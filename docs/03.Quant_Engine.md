# Quant Engine — Knowledge Transfer

## What is it?

The Quant Engine is a **deterministic analytics service** — it takes raw time-series options data from InfluxDB and computes quantitative metrics that the AI agents need to make decisions. Critically, it contains **no AI or LLM calls**. It is pure math.

The key insight: **never let the LLM compute numbers**. LLMs hallucinate math. Instead, feed the LLM pre-computed structured data and let it reason about the output.

---

## What It Computes

| Metric | What it means | Why it matters for options selling |
|--------|-------------|-----------------------------------|
| **IV Percentile** | What % of the last 30 days had IV lower than today | High IV percentile (>70%) = premiums are rich = good time to sell |
| **IV Rank** | Where current IV sits in the 30-day range (0-1 scale) | Normalizes IV to a 0-1 scale regardless of absolute IV |
| **Put/Call Skew** | Ratio of put IV to call IV | Skew >1 = market is pricing more downside risk |
| **Volume/OI Ratio** | Today's volume divided by open interest | Ratio >1.5 = unusual activity (possible insider knowledge) |
| **Unusual Activity** | Boolean flag from volume/OI ratio | Triggers a risk flag on AI recommendations |

---

## High-Level Flow

```
InfluxDB (30 days of options_data)
    │
    ▼
Quant Engine (Flux queries + NumPy math)
    │
    ├─ IV Percentile (percentile rank)
    ├─ IV Rank (min-max normalization)
    ├─ Put/Call Skew (ratio)
    └─ Unusual Activity (threshold check)
    │
    ▼
QuantSummary JSON → Options AI Agent
```

## Key Components

### 1. IV History Retrieval

[**Code: main.py:18-30**](https://github.com/utkarshmahar/stocks/blob/main/services/quant-engine/main.py#L18)

```python
async def get_iv_history(symbol: str, days: int = 30) -> list[float]:
    query = f'''
    from(bucket: "{settings.influxdb_bucket}")
      |> range(start: -{days}d)
      |> filter(fn: (r) => r._measurement == "options_data" and r.symbol == "{symbol}")
      |> filter(fn: (r) => r._field == "implied_volatility")
      |> filter(fn: (r) => r.option_type == "CALL")
      |> aggregateWindow(every: 1h, fn: mean, createEmpty: false)
    '''
```

**Flux query breakdown**:
- `range(start: -30d)`: Look back 30 days
- `filter(_measurement == "options_data")`: Only option contract data (not stock quotes)
- `filter(_field == "implied_volatility")`: Only the IV field
- `filter(option_type == "CALL")`: Only calls (to avoid double-counting)
- `aggregateWindow(every: 1h, fn: mean)`: Average IV per hour to smooth out noise

**Why hourly aggregation?** Raw data has one point per strike per poll (every 30s). That's thousands of points per day. Hourly mean gives a clean time series for percentile calculation.

**Why only CALL?** Using both calls and puts would double-count IV. Calls are conventionally used for ATM IV benchmarking.

### 2. IV Percentile Calculation

[**Code: main.py:115-125**](https://github.com/utkarshmahar/stocks/blob/main/services/quant-engine/main.py#L115)

```python
iv_percentile = sum(1 for v in iv_history if v < current_iv) / len(iv_history)
```

**What this does**: Counts what fraction of historical IV values are below the current IV.

**Example**:
- If current IV = 35%, and over the past 30 days, 85% of hourly IV readings were below 35%
- Then IV Percentile = 0.85 (85th percentile)
- This means "IV is higher than usual" → premiums are fat → good for sellers

**IV Percentile vs IV Rank**:

```python
iv_rank = (current_iv - iv_min) / (iv_max - iv_min)
```

| Metric | Formula | What it tells you |
|--------|---------|-------------------|
| **IV Percentile** | % of observations < current | "How often has IV been lower?" |
| **IV Rank** | (current - min) / (max - min) | "Where is IV in its range?" |

If IV was 20% for 29 days and spiked to 100% for 1 day:
- Current IV = 25%
- IV Percentile = 96.7% (only 1 day was higher)
- IV Rank = 6.25% (25 is barely above the 20 minimum in a 20-100 range)

Both metrics give different perspectives. The AI agent receives both.

[**Code: main.py:50-68**](https://github.com/utkarshmahar/stocks/blob/main/services/quant-engine/main.py#L50)

```python
async def get_put_call_data(symbol: str) -> dict:
    query = f'''
    ...
    |> group(columns: ["option_type"])
    |> mean()
    '''
```

The Flux query groups option data by `option_type` (CALL vs PUT), computes mean IV for each, then:

```python
put_call_skew = pc_data["PUT"] / pc_data["CALL"]
```

**Interpretation**:
- `skew = 1.0` → Puts and calls have equal IV → neutral market
- `skew > 1.0` → Puts are more expensive → market expects downside → bearish hedging demand
- `skew < 1.0` → Calls are more expensive → market expects upside → bullish speculation

**Why this matters for selling**: High put skew means put sellers can collect fatter premiums.

### 4. Unusual Activity Detection

[**Code: main.py:71-88**](https://github.com/utkarshmahar/stocks/blob/main/services/quant-engine/main.py#L71)

```python
async def get_volume_oi(symbol: str) -> dict:
    # Sums total volume and open interest across all contracts
    ...

vol_oi_ratio = vol_oi["volume"] / vol_oi["open_interest"]
unusual = vol_oi_ratio > 1.5
```

**Volume vs Open Interest**:
- **Volume**: Number of contracts traded today
- **Open Interest**: Total number of open (unsettled) contracts

**The 1.5 threshold**:
- Normal: Volume/OI < 1.0 (fewer trades than outstanding positions)
- Elevated: Volume/OI = 1.0-1.5 (active day)
- Unusual: Volume/OI > 1.5 (significantly more trading than usual → possible news, earnings, insider activity)

When `unusual_activity = true`, the AI agent includes this as a risk flag in recommendations.

## Technology Deep Dive: InfluxDB Flux Query Language

Flux is InfluxDB's functional query language. It works like a pipeline (similar to Unix pipes):

```
from(bucket) |> range() |> filter() |> aggregateWindow() |> last()
```

Each step transforms the data stream:

| Operation | What it does | SQL equivalent |
|-----------|-------------|----------------|
| `from(bucket)` | Select the database | `FROM table` |
| `range(start: -30d)` | Time window filter | `WHERE time > NOW() - 30d` |
| `filter(fn: (r) => ...)` | Row-level filter | `WHERE column = value` |
| `aggregateWindow(every: 1h, fn: mean)` | Time-bucket aggregation | `GROUP BY time_bucket('1h')` |
| `last()` | Most recent value | `ORDER BY time DESC LIMIT 1` |
| `pivot()` | Reshape columnar → wide | (restructure result set) |
| `group(columns: [...])` | Group by tags | `GROUP BY column` |
| `sum()` / `mean()` | Aggregate functions | `SUM()` / `AVG()` |

**InfluxDB data model**:
```
measurement,tag1=val1,tag2=val2 field1=value1,field2=value2 timestamp
options_data,symbol=NVDA,option_type=CALL implied_volatility=0.28 1709568000000000000
```

- **Measurement** = like a table name (`options_data`, `stock_quote`)
- **Tags** = indexed metadata (fast to query: `symbol`, `option_type`, `expiration`)
- **Fields** = actual data values (not indexed: `implied_volatility`, `delta`, `bid`)
- **Timestamp** = nanosecond precision UNIX timestamp
