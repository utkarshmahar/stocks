# Ingestion Service — Knowledge Transfer

## What is it?

The ingestion service is the **data pipeline** that constantly polls the Schwab API for live options chain data and pushes it to two destinations:

1. **InfluxDB** (time-series database) — for historical analysis & charting
2. **Redis pub/sub** — for real-time WebSocket streaming to the frontend

---

## High-Level Flow

```
Schwab API (options chains + underlying quotes)
    ↓
Ingestion Service polls every N seconds
    ↓
    ├→ InfluxDB (write time-series points)
    └→ Redis pub/sub (broadcast to WebSocket subscribers)
```

### 4. Convert to InfluxDB Line Protocol

[**Code: main.py:58-119**](https://github.com/utkarshmahar/stocks/blob/main/services/ingestion-service/main.py#L58)

```python
def build_line_protocol(symbol: str, data: dict) -> list[str]:
    # Example output:
    # stock_quote,symbol=NVDA price=150.25,bid=150.20,ask=150.30,volume=12345i 1234567890000000000
    # options_data,symbol=NVDA,option_type=CALL,expiration=2025-03-21,strike=150 bid=2.50,ask=2.60,iv=0.28,delta=0.55 1234567890000000000
```

**Why line protocol?**
- InfluxDB's native wire format (fast, efficient)
- Each line = one time-series point with **tags** (indexed) and **fields** (values)

**Example breakdown:**
- `stock_quote` = measurement (like a table)
- `symbol=NVDA` = tag (indexed for queries)
- `price=150.25` = field (the actual value)
- `1234567890000000000` = nanosecond timestamp

**Two measurement types:**

1. **`stock_quote`** — underlying stock price, bid/ask, volume, Greeks
2. **`options_data`** — each option contract (bid, ask, IV, delta, gamma, theta, vega, rho, DTE, ITM flag)

[**Code: main.py:122-170**](https://github.com/utkarshmahar/stocks/blob/main/services/ingestion-service/main.py#L122)

```python
def parse_options_for_redis(symbol: str, data: dict) -> dict:
    return {
        "symbol": "NVDA",
        "quote": { "price": 150.25, "bid": 150.20, ... },
        "calls": [ { "strike": 150, "expiration": "2025-03-21", "iv": 0.28, ... }, ... ],
        "puts": [ ... ],
        "timestamp": "2025-03-04T16:30:00.123456"
    }
```

**Why separate formats?**
- **InfluxDB** = optimized for time-series storage & historical queries
- **Redis** = lightweight JSON snapshot for **real-time pub/sub** (WebSocket subscribers)
- The Redis format is what the frontend gets pushed to via WebSocket

### 6. Main Loop

[**Code: main.py:173-209**](https://github.com/utkarshmahar/stocks/blob/main/services/ingestion-service/main.py#L173)

```python
async def ingestion_loop():
    client = get_schwab_client()  # Auth once at startup

    while True:
        symbols = await get_watchlist_symbols()  # Reload each iteration
        redis = await get_redis()

        for symbol in symbols:
            # 1. Query Schwab
            data = query_options_chain(client, symbol)

            # 2. Write to InfluxDB
            lines = build_line_protocol(symbol, data)
            await influx_write(lines)

            # 3. Publish to Redis (WebSocket subscribers)
            snapshot = parse_options_for_redis(symbol, data)
            await redis.publish("options:updates", json.dumps(snapshot))

        await asyncio.sleep(settings.query_interval)  # e.g., 30 seconds
```

**Key traits:**
- **Infinite loop** — runs forever (until service crashes/restarts)
- **Polls in sequence** — if you have 10 stocks, it queries each one, pauses, repeats
- **Async/await** — non-blocking, so can handle multiple I/O operations
- **Error resilient** — one failed symbol doesn't stop others
- **Graceful fallback** — if Postgres is down, uses env variable watchlist

## Data Flow Diagram

```
Schwab API
    ↓
Ingestion Loop (every 30s)
    ├─ For each symbol:
    │  ├─ query_options_chain()    → raw JSON
    │  ├─ build_line_protocol()    → InfluxDB format
    │  │   ├─ influx_write()        → InfluxDB
    │  │
    │  └─ parse_options_for_redis() → JSON snapshot
    │      └─ redis.publish()      → Redis
    │
    ├─ InfluxDB (time-series store)
    │   └─ Used by: Quant Engine, Frontend charts
    │
    └─ Redis pub/sub (real-time)
        └─ Subscribed by: API Gateway → WebSocket → Frontend Dashboard
```

  - `REDIS_URL` — pub/sub destination
  - `SYMBOLS_LIST` — fallback tickers (comma-separated)
  - `QUERY_INTERVAL` — polling frequency (default 30s)

## Why This Design?

| Aspect | Why |
|--------|-----|
| **Two outputs (InfluxDB + Redis)** | InfluxDB for analysis; Redis for real-time WebSocket (low latency) |
| **Async/await** | Non-blocking I/O; handle multiple tickers efficiently |
| **Watchlist from Postgres** | Users can dynamically add/remove stocks without restarting |
| **Shared auth** | `get_schwab_client()` from shared module; manages token refresh |
| **Line protocol** | Native InfluxDB format; 10x faster than JSON writes |
| **Error resilience** | One bad stock doesn't break the whole loop; logs and continues |
